Yes — this is buildable now, and you do not need a graph database for the map viewer itself. GitHub Pages will happily host a static HTML/CSS/JS app, and PID officially publishes both a daily GTFS schedule feed and real-time vehicle positions plus delays through GTFS Realtime and the Golemio API. PID’s own public site also advertises a live web map covering metro, trams, trolleybuses, buses, trains, and ferries, so the data pipeline plainly exists today.

The clean design is this: keep the display as a schematic map, meaning diagram-like rather than geographically exact, and precompute the geometry, meaning calculate it ahead of time during build. PID’s GTFS docs say shape_dist_traveled in stop_times.txt has a counterpart in shapes.txt, and public Golemio examples show live vehicle payloads carrying trip.gtfs.trip_id, shape_dist_traveled, and delay fields. That is enough to map a real vehicle onto a hand-stylized metro/tram line without any routing engine behind it.

The APIs worth targeting are:

    Static network truth: data.pid.cz/PID_GTFS.zip for routes, trips, stops, and shapes; PID says it is updated daily, and Golemio also advertises a “Public GTFS Static Lookup (v2)” API.

    Live vehicle positions in JSON/GeoJSON: Golemio documents “Public Vehicle Positions (v2)” as realtime positions of vehicles in custom GeoJSON, and public examples use https://api.golemio.cz/v2/vehiclepositions.

    Live GTFS-RT feed: Golemio documents “GTFS Realtime (v2)” protobuf feeds and says they are updated every 20 seconds.

    Stop-centric delays and departures: the current docs advertise “PID Departure Boards (v4)”, which is useful if you want stop popups or platform countdown panels in addition to moving vehicles.

The main constraint for a pure GitHub Pages deployment is authentication, not rendering. Golemio’s public API access is secured with a JWT/generated API key after free registration, so any key you put into browser JavaScript is effectively public. I could not verify CORS behavior from the docs alone, so that needs an early browser test. If direct browser access fails, the static-friendly fallback is to keep the site on Pages but fetch/cache data with GitHub Actions; GitHub documents that scheduled workflows can run as often as every 5 minutes, which is fine for snapshots but not for true “live” 20-second tracking.

For the front end, MapLibre GL JS is a strong fit because it renders interactive maps in the browser with WebGL. PMTiles can also be read directly in the browser with MapLibre, but PMTiles depends on HTTP Range Requests, and I found recent community reports of GitHub Pages serving problematic range responses for that pattern in some browsers. GitHub Pages also has a published 1 GB site-size limit. So for a production-quality Pages deployment, I would avoid hosting a large PMTiles archive on Pages itself; a lighter SVG/GeoJSON schematic is safer.

My verdict: yes, feasible today. The only serious uncertainty is whether Golemio’s live endpoints will permit your exact browser-origin + exposed-key setup; if not, you still keep the project “mostly static” by adding only a tiny proxy or scheduled snapshot job.

One last legal note: PID says data downloaded directly from its site is CC-BY, while linked external sources follow their own licenses; if you use an OSM-derived basemap such as Protomaps, OSM attribution is also required.
